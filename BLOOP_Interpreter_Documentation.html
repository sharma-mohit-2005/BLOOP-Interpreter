1. Introduction & Project Overview

This project implements a complete interpreter for BLOOP (Beginner-Level Object-Oriented Program) — a small, made-up programming language that reads almost like plain English sentences. The interpreter is written entirely in pure Java with no external libraries.

When a user writes code in BLOOP and runs it through our interpreter, the program reads the source code, understands its structure, and executes it to produce real output on screen.

What BLOOP Looks Like

put 10 into x              -- assignment: store 10 in variable x
put 3 into y               -- assignment: store 3 in variable y
put x + y * 2 into result  -- arithmetic with operator precedence
print result               -- output: prints 16

if score > 50 then:        -- conditional block
    print "Pass"

repeat 4 times:            -- loop block
    print i
    put i + 1 into i

Key Features

  • Variables — store and retrieve numbers and strings
  • Arithmetic — addition, subtraction, multiplication, division with correct operator precedence
  • String output — print literal strings and string variables
  • Conditionals — if-then blocks with comparison operators (>, <, ==)
  • Loops — repeat blocks that run a fixed number of times
  • Error handling — helpful runtime error messages with line numbers

2. Architecture & Interpreter Pipeline

The interpreter works as a three-stage pipeline. Each stage takes the output of the previous one and hands its result to the next.

Source Code (.bloop file)
→
Stage 1: Tokenizer String → List<Token>
→
Stage 2: Parser List<Token> → List<Instruction>
→
Stage 3: Executor List<Instruction> → Output

Stage 1 — Tokenizer (Lexical Analysis)

The Tokenizer reads the raw source code as a single String and breaks it into a flat list of labelled pieces called tokens — like splitting a sentence into individual words. Each token carries its type (keyword, number, operator, etc.), its raw text, and the line number where it appeared.

Example: The source line put 10 into x becomes:

[PUT("put"), NUMBER("10"), INTO("into"), IDENTIFIER("x"), NEWLINE]

Stage 2 — Parser (Syntactic Analysis)

The Parser reads the token list and builds a structured tree of objects that captures the meaning of the program, including the correct order of operations for arithmetic. It produces a List<Instruction>, where each instruction represents one complete action.

Example: The tokens from put x + y * 2 into result become:

AssignInstruction("result", BinaryOpNode(VariableNode("x"), "+", BinaryOpNode(VariableNode("y"), "*", NumberNode(2))))

Stage 3 — Executor (Evaluation)

The Executor walks through the instruction list and executes each instruction. It uses an Environment object to store variables, and an OutputHandler to produce output. Each instruction can read variables, write variables, and produce output.

Why a Tree?

Arithmetic has structure. In the expression x + y * 2, multiplication must happen before addition. A flat list of tokens cannot capture this — but a tree can. The multiply node sits deeper in the tree than the add node, so it gets evaluated first:

        Add  (+)
       /    \
      x    Multiply  (*)
            /    \
           y      2

When we evaluate this tree from the bottom up: first y * 2 = 6, then x + 6 = 16. The tree shape handles operator precedence automatically.

4. Class-by-Class Explanation

4a. Token Layer

TokenType.java — Enum

An enum listing every kind of token the BLOOP language can produce. It has 21 entries organized into five categories:

CategoryToken Types
LiteralsNUMBER, STRING, IDENTIFIER
KeywordsPUT, INTO, PRINT, IF, THEN, REPEAT, TIMES
ArithmeticPLUS, MINUS, STAR, SLASH
ComparisonGREATER, LESS, EQUALS_EQUALS
StructuralCOLON, NEWLINE, INDENT, EOF

Token.java — Immutable Data Class

Represents one piece of source code. Each Token stores three pieces of information, all set in the constructor and never changed (immutability):

  • TokenType type — what kind of token (from the enum)
  • String value — the raw text from the source (e.g., "10", "put")
  • int line — line number for error reporting

Only getters are provided — no setters. This ensures tokens cannot be accidentally modified after creation.

Token.java — Key Code
public class Token {
    private final TokenType type;   // immutable
    private final String value;     // immutable
    private final int line;         // immutable

    public Token(TokenType type, String value, int line) {
        this.type = type;
        this.value = value;
        this.line = line;
    }

    public TokenType getType() { return type; }   // getter, no setter
    public String getValue()   { return value; }
    public int getLine()       { return line; }
}

4b. Expression Layer

Every expression in BLOOP — a number, a variable name, a calculation — is represented as a node object. All node classes implement the Expression interface, forming a class hierarchy through polymorphism.

Expression.java — @FunctionalInterface

The base interface with a single abstract method evaluate(Environment env). It is marked @FunctionalInterface, which means it can be implemented using lambda expressions — a key functional programming concept.

Expression.java
@FunctionalInterface
public interface Expression {
    Object evaluate(Environment env);
    // Returns Double for numbers, String for text
}

Functional Programming: Because Expression is a @FunctionalInterface, you can create expressions inline: Expression fortyTwo = env -> 42.0;

NumberNode.java — Literal Number

Stores a double value (e.g., 42 or 3.14). When evaluate() is called, it simply returns the stored number. The Environment is ignored since literal numbers are constant.

StringNode.java — Literal String

Stores a String value (e.g., "hello"). When evaluate() is called, it returns the stored string. Like NumberNode, it ignores the Environment.

VariableNode.java — Variable Reference

Stores the variable name (e.g., "x"). When evaluate() is called, it looks up the variable's current value in the Environment. If the variable hasn't been defined, the Environment throws a RuntimeException.

BinaryOpNode.java — Binary Operation

The most complex expression node. It stores three things: a left expression, an operator symbol ("+", "-", "*", "/", ">", "<", "=="), and a right expression.

Functional Programming — BiFunction Operator Map: Instead of a long if-else chain, BinaryOpNode uses a Map<String, BiFunction<Double, Double, Object>> to store operators as lambdas. This demonstrates generics, lambdas, and makes adding new operators trivial.

BinaryOpNode.java — Operator Map (Key Design)
// Generics: Map<String, BiFunction<Double, Double, Object>>
private static final Map<String, BiFunction<Double, Double, Object>> OPERATORS = Map.of(
    "+",  (a, b) -> a + b,          // lambda: addition
    "-",  (a, b) -> a - b,          // lambda: subtraction
    "*",  (a, b) -> a * b,          // lambda: multiplication
    "/",  (a, b) -> a / b,          // lambda: division
    ">",  (a, b) -> a > b,          // lambda: greater than → Boolean
    "<",  (a, b) -> a < b,          // lambda: less than → Boolean
    "==", (a, b) -> a.equals(b)     // lambda: equality → Boolean
);

@Override
public Object evaluate(Environment env) {
    Object leftVal = left.evaluate(env);
    Object rightVal = right.evaluate(env);
    // Look up the operator function and apply it
    BiFunction<Double, Double, Object> operation = OPERATORS.get(operator);
    return operation.apply((Double) leftVal, (Double) rightVal);
}

4c. Runtime Layer

Environment.java — Variable Store

The Environment is a simple mapping from variable names to their current values. Every instruction in the program shares one Environment instance during execution, allowing variables set by one instruction to be read by later ones.

Generics: Uses Map<String, Object> — the String key ensures type-safe variable names, while the Object value type allows storing both Double (numbers) and String (text) polymorphically.

Environment.java
public class Environment {
    // Generics: Map<String, Object>
    private final Map<String, Object> variables = new HashMap<>();

    public void set(String name, Object value) {
        variables.put(name, value);  // store or update
    }

    public Object get(String name) {
        if (!variables.containsKey(name)) {
            throw new RuntimeException("Variable not defined: " + name);
        }
        return variables.get(name);
    }
}

OutputHandler.java — Functional Interface for Output (DI)

A @FunctionalInterface with a single method print(String output). This enables dependency injection: instead of instructions calling System.out.println() directly, they receive an OutputHandler. This decouples the output mechanism from the logic.

OutputHandler.java
@FunctionalInterface
public interface OutputHandler {
    void print(String output);
}

// Usage with method reference (functional programming):
OutputHandler handler = System.out::println;  // production
OutputHandler testHandler = list::add;        // testing

Dependency Injection: By injecting OutputHandler, we can swap System.out::println (production) for a list-capturing handler (testing) without changing any instruction code.

4d. Instruction Layer

An instruction is one complete action BLOOP can perform. All instruction classes implement the Instruction interface through polymorphism.

Instruction.java — @FunctionalInterface

@FunctionalInterface
public interface Instruction {
    void execute(Environment env);
}

AssignInstruction.java

Handles put <expression> into <variable>. Stores the variable name and the expression. When executed, it evaluates the expression and stores the result in the Environment.

// put 10 + 5 into x  →  AssignInstruction("x", BinaryOpNode(10, "+", 5))
@Override
public void execute(Environment env) {
    Object value = expression.evaluate(env);  // evaluate: 10 + 5 = 15.0
    env.set(variableName, value);             // store: x = 15.0
}

PrintInstruction.java

Handles print <expression>. Uses dependency injection — receives an OutputHandler via constructor. Numbers like 16.0 are formatted as "16" (without decimal point) for clean output.

// DI: OutputHandler injected via constructor
public PrintInstruction(Expression expression, OutputHandler outputHandler) { ... }

@Override
public void execute(Environment env) {
    Object value = expression.evaluate(env);
    String output = formatOutput(value);  // 16.0 → "16"
    outputHandler.print(output);          // DI: uses injected handler
}

IfInstruction.java

Handles if <condition> then: <body>. Stores a condition expression and a List<Instruction> body (generics). When executed, evaluates the condition — if it's true, executes every instruction in the body.

// Generics: List<Instruction> for the body
private final List<Instruction> body;

@Override
public void execute(Environment env) {
    Object result = condition.evaluate(env);       // e.g., 85 > 50 → true
    if (result instanceof Boolean && (Boolean) result) {
        for (Instruction instruction : body) {
            instruction.execute(env);               // run body instructions
        }
    }
}

RepeatInstruction.java

Handles repeat <n> times: <body>. Stores the repeat count and a List<Instruction> body. Executes all body instructions count times.

@Override
public void execute(Environment env) {
    for (int i = 0; i < count; i++) {
        for (Instruction instruction : body) {
            instruction.execute(env);  // run body instructions each iteration
        }
    }
}

4e. Tokenizer (Lexer)

The Tokenizer reads raw source code as a String and produces a List<Token>. It walks through characters one at a time, recognising patterns and emitting a Token for each one.

How Tokenization Works

  1. Newlines — emit NEWLINE token, increment line counter
  2. Indentation — at line start, spaces/tabs emit INDENT token (used for block detection)
  3. Numbers — sequences of digits and '.' → NUMBER token
  4. Strings — text between "..." → STRING token (quotes are stripped)
  5. Words — alphabetic sequences are checked against a keyword map; if found → keyword token, otherwise → IDENTIFIER
  6. Operators — single characters like +, -, *, /, >, <, : → operator tokens
  7. EOF — appended at the end of the token list

Generics: Map<String, TokenType> is used for keyword lookup — a type-safe mapping from keyword text to its corresponding enum value.

Tokenizer.java — Keyword Map
// Generics: Map<String, TokenType> for keyword lookup
private static final Map<String, TokenType> KEYWORDS = Map.of(
    "put",    TokenType.PUT,
    "into",   TokenType.INTO,
    "print",  TokenType.PRINT,
    "if",     TokenType.IF,
    "then",   TokenType.THEN,
    "repeat", TokenType.REPEAT,
    "times",  TokenType.TIMES
);

private Token readWord() {
    // Read the word, then look up in the KEYWORDS map
    TokenType type = KEYWORDS.getOrDefault(word, TokenType.IDENTIFIER);
    return new Token(type, word, line);
}

4f. Parser (Recursive Descent)

The Parser reads the List<Token> and builds a List<Instruction>. It uses the recursive descent technique, where each grammar rule maps to a method that may call other methods recursively.

BLOOP Grammar

program     → instruction*
instruction → assignment | printStmt | ifStmt | repeatStmt
assignment  → PUT expression INTO IDENTIFIER NEWLINE
printStmt   → PRINT expression NEWLINE
ifStmt      → IF condition THEN COLON NEWLINE block
repeatStmt  → REPEAT NUMBER TIMES COLON NEWLINE block
block       → (INDENT instruction)+
condition   → expression ( > | < | == ) expression
expression  → term ( ( + | - ) term )*
term        → primary ( ( * | / ) primary )*
primary     → NUMBER | STRING | IDENTIFIER

Operator Precedence via Recursive Descent

The parser handles operator precedence through a three-level call chain:

MethodHandlesPrecedenceCalls
parseExpression()+ and -Lowest→ parseTerm()
parseTerm()* and /Higher→ parsePrimary()
parsePrimary()Numbers, strings, variablesHighest (atoms)—

This chain gives * and / higher priority than + and - automatically because parseTerm() binds its operands before parseExpression() sees them.

Instruction Dispatch

The parseInstruction() method looks at the current token and dispatches to the appropriate method:

private Instruction parseInstruction() {
    Token token = peek();
    switch (token.getType()) {
        case PUT:    return parseAssignment();  // put <expr> into <var>
        case PRINT:  return parsePrint();       // print <expr>
        case IF:     return parseIf();          // if <cond> then: <body>
        case REPEAT: return parseRepeat();      // repeat <n> times: <body>
        default:     throw new RuntimeException("Unexpected token...");
    }
}

Block Parsing

Blocks (indented code after then: or times:) are parsed by looking for INDENT tokens. Lines with indentation belong to the block; the first non-indented line ends it.

Dependency Injection: The Parser receives an OutputHandler in its constructor and injects it into every PrintInstruction it creates. This ensures output handling is configurable without modifying the instruction classes.

4g. Engine & Pipeline

ExecutionEngine.java — DI Interface

An interface with method execute(List<Instruction>, Environment). This is the dependency injection abstraction point — the Interpreter depends on this interface rather than a concrete class, allowing different execution strategies to be swapped in.

BloopExecutionEngine.java — Concrete Implementation

The standard execution engine that runs instructions sequentially using the Stream API:

@Override
public void execute(List<Instruction> instructions, Environment env) {
    // Functional programming: Stream API with lambda
    instructions.stream().forEach(instruction -> instruction.execute(env));
}

InterpreterPipeline.java — Function Composition

Demonstrates Function.andThen() composition — chaining the tokenize and parse stages into a single function:

InterpreterPipeline.java — Function Composition
// Generics: Function<InputType, OutputType> for each stage
private final Function<String, List<Token>> tokenizer;
private final Function<List<Token>, List<Instruction>> parser;
private final Consumer<List<Instruction>> executor;

public void run(String sourceCode) {
    // Functional programming: Function.andThen() chains stages
    Function<String, List<Instruction>> tokenizeAndParse =
        tokenizer.andThen(parser);

    List<Instruction> instructions = tokenizeAndParse.apply(sourceCode);
    executor.accept(instructions);
}

Function Composition: tokenizer.andThen(parser) creates a new function that first tokenizes the source code, then feeds the token list directly to the parser — a core functional programming pattern.

4h. Entry Points

Interpreter.java — Constructor Injection

The main interpreter class. All dependencies are injected via the constructor (DI pattern): it receives an OutputHandler and an ExecutionEngine. It never creates its own dependencies — they're provided from outside.

Interpreter.java — DI Constructor
public class Interpreter {
    private final OutputHandler outputHandler;      // DI: injected
    private final ExecutionEngine executionEngine;   // DI: injected

    // Constructor Injection — dependencies provided externally
    public Interpreter(OutputHandler outputHandler, ExecutionEngine engine) {
        this.outputHandler = outputHandler;
        this.executionEngine = engine;
    }

    public void run(String sourceCode) {
        Environment env = new Environment();

        // Functional: wrap stages as Function<> objects
        Function<String, List<Token>> tokenize = source ->
            new Tokenizer(source).tokenize();

        Function<List<Token>, List<Instruction>> parse = tokens ->
            new Parser(tokens, outputHandler).parse();  // DI: pass handler

        Consumer<List<Instruction>> execute = instructions ->
            executionEngine.execute(instructions, env);

        // Compose and run the pipeline
        new InterpreterPipeline(tokenize, parse, execute).run(sourceCode);
    }
}

Main.java — Composition Root

The CLI entry point and composition root — the single place where all dependencies are created and wired together:

public static void main(String[] args) {
    String sourceCode = Files.readString(Paths.get(args[0]));

    // ── DI WIRING ──────────────────────────
    OutputHandler outputHandler = System.out::println;  // method reference
    ExecutionEngine engine = new BloopExecutionEngine();
    Interpreter interpreter = new Interpreter(outputHandler, engine); // inject

    interpreter.run(sourceCode);
}

6. Step-by-Step Execution Walkthrough

Let us trace what happens when the interpreter runs this BLOOP program:

put 10 into x
put 3 into y
put x + y * 2 into result
print result

Step 1 — Main.java reads the file

Main.main() reads program1.bloop into a String, creates the DI wiring (OutputHandler = System.out::println, ExecutionEngine = new BloopExecutionEngine()), and calls interpreter.run(sourceCode).

Step 2 — Tokenizer produces tokens

The Tokenizer walks through the source character by character and produces:

Line 1: [PUT, NUMBER("10"), INTO, IDENTIFIER("x"), NEWLINE]
Line 2: [PUT, NUMBER("3"), INTO, IDENTIFIER("y"), NEWLINE]
Line 3: [PUT, IDENTIFIER("x"), PLUS, IDENTIFIER("y"), STAR, NUMBER("2"), INTO, IDENTIFIER("result"), NEWLINE]
Line 4: [PRINT, IDENTIFIER("result"), NEWLINE, EOF]

Step 3 — Parser builds instructions

The Parser reads the tokens and constructs four instructions:

1. AssignInstruction("x", NumberNode(10.0))
2. AssignInstruction("y", NumberNode(3.0))
3. AssignInstruction("result", BinaryOpNode(
       VariableNode("x"), "+",
       BinaryOpNode(VariableNode("y"), "*", NumberNode(2.0))
   ))
4. PrintInstruction(VariableNode("result"))

Note how y * 2 is nested deeper than x + ... in instruction 3 — this is how the tree structure ensures multiplication happens first.

Step 4 — Executor runs instructions

InstructionActionEnvironment After
1. AssignInstruction("x", 10.0)Evaluate NumberNode(10.0) → 10.0; store x = 10.0{x: 10.0}
2. AssignInstruction("y", 3.0)Evaluate NumberNode(3.0) → 3.0; store y = 3.0{x: 10.0, y: 3.0}
3. AssignInstruction("result", ...)Evaluate tree: y*2 = 6.0, then x+6.0 = 16.0; store result = 16.0{x: 10.0, y: 3.0, result: 16.0}
4. PrintInstruction("result")Evaluate VariableNode("result") → 16.0; format as "16"; printOutput: 16

Expression Tree Evaluation (Instruction 3 in Detail)

       BinaryOpNode("+")
       /              \
VariableNode("x")   BinaryOpNode("*")
      |               /           \
  env.get("x")  VariableNode("y")  NumberNode(2.0)
      |               |               |
    10.0          env.get("y")       2.0
                      |
                     3.0

Evaluation (bottom-up):
  1. VariableNode("y").evaluate() → 3.0
  2. NumberNode(2.0).evaluate()   → 2.0
  3. BinaryOpNode("*").evaluate() → 3.0 * 2.0 = 6.0
  4. VariableNode("x").evaluate() → 10.0
  5. BinaryOpNode("+").evaluate() → 10.0 + 6.0 = 16.0

8. How to Compile and Run

Prerequisites

  • Java Development Kit (JDK) 11 or higher

Compile

javac -d out -sourcepath src src/com/bloop/Main.java

This compiles all Java files and places the .class files in the out/ directory.

  • -d out — output directory for compiled classes
  • -sourcepath src — tells javac where to find source files

Run a Program

java -cp out com.bloop.Main tests/program1.bloop
  • -cp out — classpath: tells Java where to find compiled .class files
  • com.bloop.Main — fully qualified class name to run
  • tests/program1.bloop — path to the source file (passed as args[0])

Run All Test Programs

java -cp out com.bloop.Main tests/program1.bloop   # Output: 16
java -cp out com.bloop.Main tests/program2.bloop   # Output: Sitare / Hello from BLOOP
java -cp out com.bloop.Main tests/program3.bloop   # Output: Pass
java -cp out com.bloop.Main tests/program4.bloop   # Output: 1, 2, 3, 4
